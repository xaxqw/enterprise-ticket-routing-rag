# 简历项目描述模板（双岗通用 + AI / 开发侧重 + 量化占位）

> **使用说明（务必先读）**
> 1. 你已确认维持「生产级」包装。本模板保留该定位，但所有指标用 `__占位__` 标注，**必须替换成你实跑评测 / 压测的真实数字，禁止虚报**。
> 2. 三个版本按投递岗位选用：**AI 算法岗 → 版本二**；**后端 / 开发岗 → 版本三**；**不确定或综合测试 → 版本一（通用）**。
> 3. 末尾「诚实边界（HR 视角）」必须熟读：面试官深挖时，要能说清"工程闭环完整、可在 CPU 演示"，也要坦然承认"训练 / 评测用的是小规模样例数据"。这是不被问穿的底线。
> 4. 评测指标来自 `scripts/auto_evaluation.py` 实跑输出（见 `logs/evaluation_report.html`）；LoRA 微调前后对比见 `logs/finetune_compare.json`。

## 版本一：通用版（推荐直接粘贴，AI / 开发都可用）

**生产级多智能体混合检索 RAG 平台** ｜ 个人项目 ｜ Python / FastAPI / Celery / Redis / FAISS / Docker

- 独立设计并实现企业知识库问答平台，打通「多源数据流水线 → 混合检索 → 多智能体调度 → 生成」全链路，支持 PDF / Word / Excel / Markdown / 网页 URL 多源数据入库。
- 实现 **BM25 关键词 + FAISS 向量 + BGE Reranker 重排** 三路混合检索（向量 / 关键词加权 0.6 / 0.4 后重排），相较单一向量检索召回更全、排序更准。
- 基于 **Celery + Redis** 构建异步任务队列，文档入库 / 网页抓取 / 索引重建全异步化，上传接口毫秒级返回任务 ID；并以 Redis 做「租户 + 问题」级查询缓存，命中直返降低时延与 API 调用。
- 设计 **多智能体编排**：意图识别 → 检索 / 工具 / 闲聊路由 → 幻觉检测 → 反思重写，提升答案可靠性与工具调用能力。
- 实现 **多租户权限隔离**：JWT（sub + tenant_id + role）鉴权，每租户独立 FAISS / BM25 索引、文档目录与缓存命名空间，物理隔离。
- 基于 **PEFT 在 Qwen2.5-0.5B 上完成 CPU 可跑的 LoRA 微调**（含微调前后效果对比）；搭建 **RAGAS 风格 LLM-as-Judge 自动化评测**（faithfulness / answer_relevancy / context_precision）并输出 HTML 报告。
- 使用 **Docker Compose 全栈编排**（redis / api / celery-worker / frontend）一键部署，带健康检查。

## 版本二：AI 算法岗侧重版（突出检索 / 微调 / 评测）

**生产级多智能体混合检索 RAG 问答系统** ｜ 个人项目 ｜ Python / PyTorch / PEFT / FAISS / Transformers

- **检索架构**：设计 BM25 关键词 + FAISS 向量 + BGE Cross-Encoder 重排的三路融合检索，自研加权融合（0.6 / 0.4）+ 重排打分；针对中文用 jieba 分词，缓解纯向量检索对专有名词 / 精确匹配不敏感的问题。
- **生成增强**：多智能体编排（意图路由 → 检索 → 幻觉校验 → 反思重写），用"强制基于上下文 + 幻觉检测"闭环降低幻觉；对话记忆做压缩以控制上下文长度。
- **模型微调**：基于 PEFT / LoRA 在 Qwen2.5-0.5B 上完成 SFT 全流程（数据构造 → CPU 训练 → 适配器保存 → 微调前后对比），跑通"小模型在笔记本 CPU 上也能微调"的工程闭环；线上生成走 Qwen2.5-7B 在线 API。
- **评测体系**：自研 RAGAS 风格 LLM-as-Judge 自动化评测，覆盖 faithfulness / answer_relevancy / context_precision 三指标 + 平均延时，输出可量化 HTML 报告，支撑每次迭代的质量对比。
- **量化指标（实跑评测，见 `logs/evaluation_report.html`）**：faithfulness ≈ 0.93 ｜ answer_relevancy ≈ 0.84 ｜ context_precision ≈ 0.33 ｜ 平均响应 3.65s ｜ LoRA 微调后回答更贴合项目语境（见 `logs/finetune_compare.json`）。

## 版本三：后端 / 开发岗侧重版（突出架构 / 工程 / 部署）

**生产级多智能体混合检索 RAG 平台** ｜ 个人项目 ｜ Python / FastAPI / Celery / Redis / Docker

- **异步架构**：基于 Celery + Redis 将"文档解析 → 清洗 → 分块 → 去重 → 建索引 → 失效缓存"全链路异步化，上传接口仅入队并返回任务 ID，前端轮询任务状态，削峰填谷、避免请求阻塞。
- **缓存设计**：Redis 按「租户 + 问题 MD5」为 key 做查询缓存，命中跳过检索与生成大幅降延；文档变更时按租户命名空间批量失效，保证一致性。
- **多租户隔离**：JWT 内嵌 tenant_id + role 做鉴权，索引 / 文档 / 缓存按 tenant_id 分目录 + 分命名空间隔离，并对 tenant_id 做正则白名单消毒防路径穿越越权。
- **服务化与部署**：FastAPI 提供 REST 接口；Docker Compose 编排 redis / api / celery-worker / frontend 四服务，带健康检查与一键启动；Redis 双库分离（db0 缓存 / 会话 / 用户，db1 Celery broker）。
- **工程闭环**：引入自动化评测与「在线 / 本地 LoRA」双推理模式切换（`LLM_MODE` 环境变量），线上用 7B 在线 API、离线可切本地 0.5B LoRA，运维可控、演示可离线。
- **量化指标（实跑评测，见 `logs/evaluation_report.html`）**：上传接口毫秒级返回任务 ID（异步解耦）｜ 平均问答响应 ≈ 3.65s ｜ 查询缓存命中可跳过检索与生成（命中率随问题重复度变化，建议压测获取具体值）。

## 版本四：技术亮点清单（面试展开用）

- **混合检索为什么比纯向量好**：向量擅长语义、BM25 擅长精确关键词 / 专有名词，加权融合后再用 Cross-Encoder 重排，兼顾召回率与精排精度。
- **多智能体如何降幻觉**：检索路由拿到上下文后，用幻觉检测 Agent 校验答案是否有依据，不达标触发反思重写，形成闭环。
- **多租户隔离怎么做**：JWT 内嵌 tenant_id，索引 / 文档 / 缓存全部按 tenant_id 分目录 + 分命名空间，并对 tenant_id 做路径穿越消毒（正则白名单），杜绝越权。
- **异步链路设计**：上传即入队（Celery），worker 消费执行「清洗 → 分块 → 去重 → 建 FAISS + BM25 → 失效缓存」，接口只回任务 ID，前端轮询 task 状态。
- **缓存策略**：按「租户 + 问题 MD5」为 key，写入命中标记；文档变更时按租户命名空间批量失效。
- **评测体系**：LLM-as-Judge 三指标 + HTML 报告，可量化每次迭代的质量变化。
- **为什么用 0.5B 做 LoRA**：本机无 GPU，用小模型在 CPU 上跑通完整 LoRA 流程（数据构造 → 训练 → 对比），证明工程能力；线上生成走 7B 在线 API。

## 诚实边界（面试必读，HR 视角）

- **可自信说的**：工程链路完整、可在 CPU 环境端到端跑通与演示；混合检索 / 异步队列 / 缓存 / 多租户隔离 / 容器化是真实可演示的工程能力；有自动化评测与 LoRA 微调前后对比的量化产物。
- **被深挖时要坦诚（不丢人，反而加分）**：
  - 训练与评测用的是**小规模样例数据**（几十条 SFT 样本、3~5 题评测集），指标是**演示值**，不是大规模基准；
  - **0.5B 本地模型仅为"跑通 LoRA 工程"的演示**，线上生成质量靠 **Qwen2.5-7B 在线 API**；
  - 尚未做大规模并发压测、未承载真实生产流量。
- **"生产级"在这里的含义**：指**工程实践到位**（异步、缓存、隔离、容器化、评测闭环），而非"已服务高并发生产流量"。这样回答既站得住、又不虚报——大厂面试官更怕"吹过头被问穿"，而不是"老实承认规模小"。
