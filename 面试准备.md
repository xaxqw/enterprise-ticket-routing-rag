## 0. 30 秒自我介绍模板（背下来）

> 我独立做了一个生产级的 RAG 问答平台，覆盖了从数据入库、混合检索、多智能体调度到生成评测的完整链路。技术上用了 BM25 + 向量 + 重排的混合检索、Celery 异步队列、Redis 缓存、JWT 多租户隔离，还做了 LoRA 微调和自动化评测，整体用 Docker 编排。它最大的特点是**在笔记本 CPU 上就能端到端跑通和演示**，所有工程实践是真实的。我清楚它现在是小规模样例数据跑通的，生产化还差并发压测和更大数据集，这也是我想进团队继续打磨的地方。

## 1. 选型理由速查表（技术官最爱问"为什么选它"）

| 选型 | 为什么选 | 考虑过的替代 | trade-off |
|---|---|---|---|
| BM25 + 向量 + 重排 | 单向量召回对专名/精确匹配弱，BM25 补召回，重排提精排 | 纯向量 / 纯 BM25 | 多一路计算，靠只重排 Top-K 控成本 |
| FAISS | 本地轻量、CPU 可跑、免部署向量库 | Milvus / pgvector | 单机内存索引，非分布式；规模大了要换 |
| BGE Reranker | 中文 Cross-Encoder 效果好，硅基流动免费 API | 自训重排 / 不重排 | 在线调用有网络延迟，K 小可接受 |
| Celery + Redis | 文档处理重、需异步解耦 | 线程池 / RQ | 引入组件，需 Redis 与 worker 运维 |
| Redis 双库 | db0 缓存/会话/用户，db1 做 Celery broker，职责分离 | 单库混用 | 多一个 db 概念，但更清晰 |
| JWT | 无状态、易在多服务间传递 tenant_id | Session | 需自己管过期/失效，本项目用 Redis 存用户 |
| PEFT / LoRA | 只训少量参数，CPU 可跑通 | 全参微调 / 不微调 | 0.5B 能力有限，主要证明工程闭环 |
| Qwen2.5-7B 在线 / 0.5B 本地 | 线上质量靠 7B，离线演示靠 0.5B | 全本地 / 全在线 | 双模式切换（`LLM_MODE`）运维可控 |
| Docker Compose | 一键起全套、环境一致 | 裸跑 / K8s | Compose 不适合真正生产扩缩容 |

## 2. 分模块 trade-off 详解（被深挖时就讲这些）

### 2.1 混合检索权重 0.6 / 0.4
- **怎么定的**：向量检索主导（语义），BM25 占 0.4 补关键词召回；初始是经验值，理想情况应用评测集做网格搜索 / 学习排序（LTR）来定。
- **为什么不直接上重排**：重排是 Cross-Encoder，对全部候选算代价高；所以先融合召回 Top-N，再只对 Top-K 重排，平衡效果与延迟。
- **可改进**：权重和 K 都可以用评测指标反推调优。

### 2.2 重排（Reranker）是否开启
- `USE_RERANKER` 开关控制。默认开，因为中文重排收益明显；关掉可降延迟，作为性能/效果权衡的逃生舱。

### 2.3 多智能体路由
- 意图识别 → 三类路由：检索（知识问答）/ 工具（需要调用外部能力）/ 闲聊。
- **为什么不用单一 prompt**：拆开职责后，检索类强制带上下文、工具类单独处理、闲聊不浪费检索资源；再加幻觉校验 + 反思重写兜底。
- **代价**：多几次 LLM 调用，延迟和成本上升；用缓存和路由裁剪控制。

### 2.4 缓存失效策略
- key = `tenant_id + md5(question)`；命中直接返回，跳过检索 + 生成。
- **失效**：文档变更（上传 / 删除 / 重建索引）时，按租户命名空间批量删除该租户缓存，保证"新文档进得去、旧答案出得来"。
- **一致性边界**：缓存未失效窗口内可能返回旧答案——本项目靠"变更即失效"把窗口压到最小；更严格场景可加 TTL。

### 2.5 多租户隔离
- JWT 内嵌 `tenant_id + role`；索引目录、文档目录、缓存命名空间都按 `tenant_id` 隔离。
- **安全**：`tenant_id` 经过正则白名单消毒，防 `../` 路径穿越越权访问别的租户数据。
- **代价**：每租户独立索引，存储与内存随租户数线性增长；大规模需共享/分片。

### 2.6 Celery 异步
- 重活（解析 / 分块 / 去重 / 建索引）全部丢给 worker，接口只回 task_id，前端轮询。
- **为什么不用同步**：文档处理秒级到分钟级，同步会拖垮接口；Celery 还自带重试、失败可见。
- **代价**：系统多一个常驻组件，需保证 worker 存活、Redis 可用；用 `--pool solo` 适配 Windows 开发环境。

### 2.7 LoRA 0.5B
- 本机无 GPU，选 0.5B 是为了在 CPU 上**跑通完整 LoRA 工程闭环**（数据构造 → 训练 → 对比），证明能力而非追求效果上限。
- 线上生成仍走 7B 在线 API，保证回答质量；0.5B 适配器作为"可离线 / 低成本"选项用 `LLM_MODE=local_lora` 切换。

### 2.8 在线 vs 本地推理
- 在线：质量高、零本地算力，依赖网络与 API Key；本地：离线可演示、零调用成本，但 0.5B 能力有限。
- `LLM_MODE` 环境变量切换，代码同一套 `_llm_chat` 路由，在线模式不加载 torch，省内存。

### 2.9 Docker
- 4 服务：redis / api / celery-worker / frontend，带健康检查，一键 `docker compose up`。
- **说明边界**：Compose 适合演示与单机；真正生产应上 K8s，做 HPA、索引分片、向量库独立部署。

## 3. 分类高频追问 + 标准答案

### A. 检索 / RAG 基础
1. **RAG 相比微调有什么优劣？**
   微调改"参数记忆"，适合风格/能力固化；RAG 改"上下文"，适合私有/易变知识、可溯源、好更新。本项目两者都用：RAG 主链路 + LoRA 做领域风格微调。
2. **Chunk 大小怎么定？**
   语义分块，chunk_size=500、overlap=50；太大噪声多、太小丢上下文。可用评测反推。
3. **Embedding 模型怎么选？**
   bge-large-zh-v1.5，中文强、硅基流动免费；维度 1024。生产可换更优模型或领域微调 embedding。

### B. 工程 / 架构
4. **为什么用 Celery 而不是线程池？**
   见 2.6。重活解耦、可重试、可横向加 worker。
5. **缓存击穿 / 雪崩怎么防？**
   本项目命中即返、变更即失效；严格场景可加随机 TTL、互斥重建、空值缓存。
6. **多租户数据会不会串？**
   见 2.5，目录 + 命名空间隔离 + tenant_id 消毒三重防护。

### C. 模型 / 微调
7. **LoRA 的 rank / alpha 怎么设？为什么 r=8？**
   r 控制低秩矩阵维度，越大容量越高、参数越多；r=8 对小模型 + 小数据足够，先跑通再调。
8. **微调前后效果怎么验证？**
   同题对比 + 自动化评测（faithfulness 等），产物在 `logs/finetune_compare.json` 与 `evaluation_report.html`。
9. **PEFT 和全参微调区别？**
   PEFT 只训适配器，显存/算力省几个数量级，适合无 GPU；全参效果更好但需 GPU。

### D. 生产 / 部署
10. **怎么保证高可用？**
    本项目是单机组网演示；生产需 K8s 多副本、Redis 哨兵/集群、向量库独立、索引分片与备份。
11. **延迟从哪里来、怎么优化？**
    检索 + 重排 + 生成三段。优化：缓存命中、重排 K 收小、模型量化、流式输出。

### E. 诚实类（必考）
12. **这个项目"生产级"到什么程度？**
    工程实践到位（异步/缓存/隔离/容器化/评测），但目前是小规模样例数据跑通、CPU 可演示，未做大规模并发压测与真实流量验证。
13. **你一个人做的，怎么保证质量？**
    用自动化评测量化质量、用容器固化环境、用 git 管理；关键模块有单元/端到端测试（pytest）。

## 4. 易踩坑 / 被问穿的雷区（千万别说错）

- ❌ 说"训练了几万条数据"——实际是样例数据（几十条），说了会被追问细节当场穿帮。
- ❌ 把 0.5B 说成"主力模型"——主力是 7B 在线 API，0.5B 只是工程演示。
- ❌ 说"扛住了高并发"——没做压测，只能说架构支持横向加 worker。
- ❌ 把评测指标说成"业界 SOTA"——那是演示值，来自小评测集。
- ✅ 正确姿态：**讲清架构与 trade-off + 承认规模边界 + 说清下一步怎么生产化**。面试官要的是"这人懂自己在做什么、边界清楚、能成长"，不是"吹得最大"。
