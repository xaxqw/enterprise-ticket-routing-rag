## 1. 总架构（能白板画出来）

```
Streamlit 前端(8501)  --HTTP+JWT-->  FastAPI 后端(8000)
                                   ├─ 多智能体编排（意图/路由/幻觉/反思）
                                   ├─ 混合检索（BM25 + FAISS向量 + RRF重排）
                                   └─ Celery + Redis 异步（入库/抓取/重建）
                                         │
                                    Redis（缓存 + Celery）
                                         │
                            本地 Ollama（bge-m3 向量化 / qwen2.5:7b 生成，GPU）
```

一句话讲清数据流：**问题 → 意图路由 → 混合检索召回片段 → RAG 生成答案 → 幻觉检测 →（不达标则）反思重检索 → 返回**。

## 2. 模块级 Q&A（重点必考）

### 2.1 混合检索（最高频）
- **Q：为什么既要 BM25 又要向量检索？**
  A：向量抓语义相似，BM25 抓关键词/专有名词（如产品名、型号、编号），两者互补。评测 ablation 直接证明了这点——纯向量 `recall@5=0.944`、纯 BM25 `=1.0`、混合 `=1.0`。
- **Q：α=0.6 / 0.4 怎么定的，拍脑袋吗？**
  A：**数据驱动**。我跑了 `weight_sweep`，α 从 0 到 1 步长 0.1：在 α∈[0,0.9] 区间 `recall@5` 全是 1.0，只有纯向量（α=1.0）掉到 0.944。我选向量主导（0.6）是因为在 `recall@1`/`MRR` 上向量更稳（纯向量 MRR=0.907，混合=0.921），且向量负责语义召回。所以这个权重是 ablation 选出来的，不是拍的。
- **Q：为什么用 RRF 重排而不是交叉编码器（BGE）重排？**
  A：RRF（倒数排名融合）对两套检索的分数量纲不敏感、零成本、离线、无需额外模型；本场景 baseline 已经 `recall@5=100%`，重排是锦上添花。cross-encoder 重排作为**可选升级路径**已在 `.env`（`RERANKER=cross_encoder`）预留，但默认不开，因为收益不抵成本。

### 2.2 向量库
- **Q：为什么用 FAISS？为什么不直接上 Milvus / 向量数据库？**
  A：单机本地部署、语料在百万级以内，`IndexFlatIP` 暴力检索精度就是召回上限，完全够；Milvus 是分布式/高并发场景才需要，引入运维成本不划算。
- **Q：检索为什么用 numpy 矩阵而不是 `faiss.search`？**
  A：向量已 L2 归一化，**numpy 精确内积 = 余弦相似度，与 FAISS IndexFlatIP 数学等价**。用内存矩阵能让检索、重排、融合共享同一份向量，避免对 FAISS 索引的重复访问；FAISS 负责索引的构建与持久化。核心论点：**数学等价，不是偷工减料**。（⚠️ 面试时只讲"数学等价 + 共享矩阵"，绝对不要说"为了技术叙事"这种词。）

### 2.3 多智能体编排
- **Q：你的系统真的是"Agent"吗？**
  A：是 Agentic RAG。有：意图路由（`AgentOrchestrator`）、检索 Agent、工具 Agent（计算器/时间）、幻觉检测 Agent、以及反思重检索循环（`process_with_reflection`）。
- **Q：意图路由为什么用规则（关键词+正则）而不是 LLM 路由？**
  A：规则路由**延迟毫秒级、零成本、完全可解释、可控**。中文意图（知识问答 / 计算 / 闲聊）用关键词+正则已能稳定覆盖主要场景。LLM 路由适合更开放的域，我把它作为**可选升级路径**写进设计，但默认不开——因为本项目不需要为确定性场景付出 LLM 路由的延迟和成本。

### 2.4 幻觉检测
- **Q：怎么防幻觉？**
  A：`HallucinationAgent` 做句子级事实校验：先过滤"根据参考资料…""注意…"这类套话/免责声明，再对实质性陈述用 jieba 分词算**关键词覆盖率**和**事实覆盖率**；任一低于 0.5 就判为不可信，触发反思重检索。也预留了 LLM-NLI 作为更强校验（默认不开）。

### 2.5 本地免费运行
- **Q：为什么全本地、不调用云端大模型？**
  A：三个理由——**零成本**（无 token 费用）、**隐私**（企业知识不出本机）、**离线可用**。技术栈：bge-m3 做向量化（1024 维，多语言/中文强），qwen2.5:7b 做生成，自动调用 RTX 4050 GPU。在线 SiliconFlow 仅作为可选后端保留。

## 3. 高频追问 TOP 10（预先背熟）

1. **FAISS 在千万/亿级会怎么崩？** → `IndexFlatIP` 是 O(N) 暴力检索，亿级要换 IVF-PQ / HNSW 或上 Milvus；我这是百万级单机场景，暴力检索精度最优。
2. **你的 RAG 和 LangChain 比？** → 我没用 LangChain，自己编排。理由：可控、无黑盒、依赖少、好调试；也认可 LangChain 适合快速原型，工程深度够了我选择自研。
3. **7B 模型够吗？生成质量如何？** → 见第 4 节（诚实讲弱点）。
4. **多租户怎么隔离？** → JWT（sub+tenant_id+role）鉴权；每租户独立的 FAISS/BM25 索引目录、文档目录、缓存命名空间，物理隔离。
5. **缓存怎么设计？** → Redis，按「租户+问题」维度缓存，命中直接返回，显著降低延迟。
6. **为什么 Celery + Redis 做异步？** → 入库/URL 抓取/索引重建是重 IO，异步避免阻塞 API，接口秒回 task_id。
7. **端到端延迟多少？** → 评测平均 ~14s，瓶颈在 7B 生成；检索本身是毫秒级。
8. **怎么评测？** → 检索层 ablation（确定性，无需 LLM）+ 生成层 RAGAS 风格 LLM 裁判（事实召回 / 忠实度 / 答案相关性 / 上下文精度）。
9. **你的评测指标好看吗？** → 检索 `recall@5=100%`、`MRR=0.935`；生成忠实度 0.756、相关性 0.95；弱点见下。
10. **如果上线，生成层你会怎么优化？** → 换更大本地模型或切可选在线 backend、强制 citation 引用溯源、上下文压缩、query rewrite。

## 4. 诚实的弱点 & 防御（最重要，背熟）

**真实评测数字（来自 `logs/evaluation_report.json`，18 题）：**

| 指标 | 数值 | 含义 |
|---|---|---|
| retrieval recall@5 | **1.0** | 检索召回已解决 |
| retrieval MRR | **0.935** | 首条命中质量高 |
| fact_recall（生成事实召回） | **0.454** | 7B 模型偶尔漏子事实 |
| accuracy（关键事实全覆盖率） | **0.278** | 仅 27.8% 题"所有关键事实"都被覆盖 |
| faithfulness（忠实度） | **0.756** | 多数回答基于原文 |
| answer_relevancy（相关性） | **0.95** | 回答切题 |
| context_precision（上下文精度） | **0.2** | 召回 5 条里常用到 1 条 |

**怎么讲（这是 senior 行为，反而加分）：**
> "我的评测显示**检索层已经做到 recall@5=100%，问题不在召回，而在 7B 本地模型的生成保真度**——它偶尔会漏掉某个子事实或改写原文（fact_recall 0.454、关键事实全覆盖率 0.278）。这正是我在架构里加**幻觉检测 Agent + 反思重检索**的原因：生成后做事实校验，不达标就带着原问题重新检索。同时我把在线 LLM backend 作为可选路径预留，高价值 query 可以切更强模型。这是**成本 / 质量 / 隐私**的 deliberate tradeoff，不是架构缺陷——而且我是用评测数据定位到这个瓶颈的。"

⚠️ **绝对不要**主动吹 accuracy；把它讲成"我通过评测定位到瓶颈在生成层，并设计了缓解路径"。这比指标本身更打动面试官。
